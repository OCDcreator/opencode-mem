# 出问题时先看哪里

如果你发现插件“好像没工作”，最有用的第一步通常不是猜，而是看日志。

## 哪些情况适合先看日志

- 自动采集一直没生成新记忆
- 文档页面或 WebUI 打不开
- 搜索结果明显不对
- Provider 或嵌入模型初始化失败

## 日志在哪里

默认日志文件会写到：

`~/.opencode-mem/opencode-mem.log`

如果你自己改过环境变量，也可能写到你指定的新位置。

## 项目级配置时，这里要特别注意

很多人会以为“我已经用了项目级配置，所以日志也应该在项目里”。

当前这个仓库不是这样。

默认情况下：

- 配置可以是项目级覆盖
- 数据也可以通过 `storagePath` 改成项目级
- 但日志路径仍然默认是全局的

也就是：

`~/.opencode-mem/opencode-mem.log`

如果你想让某个项目把日志写到项目目录里，需要额外设置环境变量：

`OPENCODE_MEM_LOG_FILE`

## 日志里通常能看到什么

- 插件有没有成功启动
- 自动采集有没有触发
- 模型或 Provider 有没有报错
- 某一步是成功了，还是卡住了

## 一个实用的排查顺序

1. 先看插件有没有正常启动
2. 再看你关心的功能有没有触发
3. 最后看具体报错出现在什么阶段

## 需要注意的事

日志里可能会包含提示词片段、模型输出片段，甚至部分本地路径信息。

所以：

- 自己排查时可以直接看
- 如果要发给别人，最好先删掉敏感内容

日志不是给日常使用看的，但在“为什么没生效”这个问题上，它往往是最快的答案来源。
