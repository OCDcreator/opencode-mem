# Setup Guide: The Fastest Way to Configure It

If you want the plugin working quickly, this is the page to follow.

## Step 1: Open the config file

The global config file is usually here:

`~/.config/opencode/opencode-mem.jsonc`

If the file does not exist yet, the plugin usually creates a template for you.

If you only want to override settings for one project, you can also create:

`.opencode/opencode-mem.jsonc`

That project-level file overrides the global one.

## Step 2: Start with a recommended config

Use a minimal working setup like this:

```jsonc
{
  "webServerEnabled": true,
  "webServerPort": 4747,

  "autoCaptureEnabled": true,

  "opencodeProvider": "zhipuai-coding-plan",
  "opencodeModel": "glm-4.5",

  "embeddingModel": "text-embedding-v4",
  "embeddingDimensions": 1024,
  "embeddingApiUrl": "https://dashscope.aliyuncs.com/compatible-mode/v1",
  "embeddingApiKey": "env://DASHSCOPE_API_KEY",
}
```

If you use a different provider, replace the provider and model with your own values.

## If you want to keep using a local embedding model

That is still supported.

You can configure embeddings like this:

```jsonc
{
  "embeddingModel": "Xenova/nomic-embed-text-v1",
}
```

Or combine it with the rest of your setup:

```jsonc
{
  "webServerEnabled": true,
  "webServerPort": 4747,

  "autoCaptureEnabled": true,

  "opencodeProvider": "zhipuai-coding-plan",
  "opencodeModel": "glm-4.5",

  "embeddingModel": "Xenova/nomic-embed-text-v1",
}
```

This means:

- auto capture and profile analysis still use your chosen provider
- search embeddings are generated by a local embedding model

## If you want auto capture to use a local endpoint

This is configured through the manual API settings, not through `embeddingModel`.

For example:

```jsonc
{
  "memoryProvider": "openai-chat",
  "memoryModel": "your-local-model-name",
  "memoryApiUrl": "http://127.0.0.1:11434/v1",
  "memoryApiKey": "ollama",
}
```

## Step 3: Understand the most important settings

### `opencodeProvider` and `opencodeModel`

These decide which model the plugin uses for auto capture and user profile analysis.

### `embeddingModel`, `embeddingApiUrl`, and `embeddingApiKey`

These decide how search embeddings are generated.

### `webServerPort`

This controls where the WebUI is served. The usual default is `4747`.

### `storagePath`

This controls where the plugin stores databases, vector data, and profile data.

The default is global:

`~/.opencode-mem/data`

If you want one project to keep its own separate data, set a project-local path, for example:

```jsonc
{
  "storagePath": ".opencode/opencode-mem-data",
}
```

## Step 4: Restart the current project instance

After changing config, restart the current OpenCode project instance so the new settings are picked up.

## Step 5: Verify that it worked

Check in this order:

1. Can `http://127.0.0.1:4747` open?
2. Does the Project Memories page load normally?
3. Do new memories appear after a while?
4. Does the User Profile page eventually show content?

## Global config vs project-level config

### Global config

If you only edit:

`~/.config/opencode/opencode-mem.jsonc`

then the usual result is:

- config is global
- stored data is global by default
- logs are global by default

### Project-level config

If you add:

`.opencode/opencode-mem.jsonc`

then that project can override the global settings.

Common project-level overrides are:

- `opencodeProvider`
- `opencodeModel`
- `embeddingModel`
- `storagePath`

### One important difference

Project-level config does not automatically make logs project-level.

In the current repo:

- stored data can be made project-local through `storagePath`
- logs still default to the global file `~/.opencode-mem/opencode-mem.log`

If you want project-specific logs too, set:

`OPENCODE_MEM_LOG_FILE`

## One important reminder

If you use `text-embedding-v4`, make sure you also set:

`"embeddingDimensions": 1024`

That value matters. Missing it can lead to search or indexing problems.
